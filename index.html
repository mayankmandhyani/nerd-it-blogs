// CONTRIBUTOR GUIDE

Write a post

NerdIT Blog publishes work from the community, not just the core team. Here's the bar, and the actual process.

The bar

  • No AI slop. Write from something you actually built or broke.
  • Show the failure. The dead end is usually more useful than the clean win.
  • Stack transparency. Name the actual tools and versions you used.
  • Open build logs. Link the repo or live project if one exists.
What gets rejected

Generic listicles, reworded documentation, anything written to rank rather than to help someone stuck at 1am.

How to actually submit one

  1. Pitch it first. A few sentences on what you built is enough.
  2. Duplicate an existing post. Same structure, so the TOC and code blocks keep working.
  3. Send it back. Reviewed against the bar above, published with your byline.

Get in touch

Send a pitch or draft to blog@nerdit.in.

Shipping an AI agent SMBs actually use

Claude-powered email drafting, WhatsApp follow-ups, and the role-based auth nobody asks for until launch day.

Mayank · June 2, 2026 · NerdIT community
TL;DR

Small businesses don't want "an AI agent." They want fewer emails to write and to not worry about staff seeing something they shouldn't. Everything below is what it actually took to ship that.

Most "AI for SMBs" demos show the same three minutes: type a prompt, get a polished email, close the laptop. Nobody demos the part where the owner asks "can my staff see customer replies?" and you realize you haven't built auth yet.

The actual brief

Three things the agent needed to do, in order of how often they'd actually get used:

  • Draft replies to incoming customer emails, ready to review and send — not auto-sent.
  • Follow up on leads gone quiet, via WhatsApp/SMS copy the owner pastes and sends.
  • Keep roles separate — Owner sees everything; Employee sees only what they're assigned.

Drafting that doesn't sound like a robot

The naive version is one system prompt and a temperature knob. It falls apart once two different businesses use it. The fix was giving the model real examples of the business's past replies as context, so tone came from real data instead of a generic instruction.

agent/draftReply.js
async function draftReply({ incomingEmail, businessProfile, pastReplies }) {
  const response = await anthropic.messages.create({
    model: "claude-sonnet-4-6",
    max_tokens: 600,
    system: buildSystemPrompt(businessProfile, pastReplies),
    messages: [{ role: "user", content: incomingEmail.body }]
  });
  return { draft: response.content[0].text, status: "pending_review" };
}

Every draft goes into a review queue — nothing sends itself. That one decision solved the trust problem better than any prompt engineering.

The feature owners actually asked for wasn't smarter AI. It was a send button they still control.

Role-based auth nobody budgets for

This ate more time than the AI integration. A staff member seeing a customer's full complaint history when they were only meant to handle today's queue isn't a UI bug — it's a privacy incident.

middleware/requireRole.js
function requireRole(...allowedRoles) {
  return (req, res, next) => {
    if (!allowedRoles.includes(req.user.role)) {
      return res.status(403).json({ error: "Not authorized" });
    }
    next();
  };
}
router.get("/billing", requireRole("owner"), getBillingSummary);

What broke on launch day

The first Vercel deploy failed because the project was raw HTML instead of a Vite build. Identical locally, confusing routing errors in production. Now the first thing checked before any client deploy.

What I'd do differently

Build the role-based auth before the AI drafting, not after. It's less exciting, which is exactly why it's easy to postpone — and exactly why postponing it is the mistake.

What nobody tells you about the Meta Ads API

Saved Audiences can't be created programmatically. Neither can your assumptions survive contact with the docs.

Mayank · May 24, 2026 · NerdIT community
TL;DR

Meta's Marketing API can do most of what Ads Manager does — except a few specific things it flatly can't, and the docs don't lead with that.

The plan: a small internal tool that spins up campaigns and audiences without touching Ads Manager directly. Most of it worked first try. The parts that didn't are worth writing down.

Saved Audiences are not API-creatable

Ads Manager lets you save a targeting spec as a reusable "Saved Audience." The Marketing API has no endpoint for creating one. Custom Audiences (customer lists, lookalikes) work fine via API — Saved Audiences are a UI-only construct.

response · POST /act_<id>/saved_audiences
{
  "error": {
    "message": "Unsupported post request. Object does not exist,
                cannot be loaded, or does not support this operation.",
    "type": "GraphMethodException",
    "code": 100
  }
}

The fix isn't a fix: pass the targeting object inline on every ad set instead of referencing a saved one.

Audience typeAPI-creatable?
Custom Audience (customer list)Yes
Lookalike AudienceYes
Saved Audience (demographics)No — inline only

Under-18 targeting is a policy wall, not a bug

A campaign for 16–18 year-olds kept failing with a vague targeting error. Not malformed — Meta restricts under-18 targeting platform-wide, enforced identically via API and UI. No JSON tweak fixes a policy rejection.

Half of debugging an ads API is telling "I did this wrong" apart from "this isn't allowed, and the error just doesn't say so."

Business Manager is not optional

Serious API usage assumes the ad account sits inside a Business Manager. Personal accounts hit permission walls on things that look unrelated — token generation, system users, some audience endpoints.

What I'd check next time

  • Ad account is inside a Business Manager — checked before writing any code.
  • Audience logic plans around inline targeting specs, not Saved Audience references.
  • Age-restricted campaigns get their range confirmed against policy before build time.
